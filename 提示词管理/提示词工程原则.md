# 提示词工程原则

行为指南，用于减少提示词写作的常见失败模式。源自 Karpathy 编码四原则 + 2025-2026 提示词工程最佳实践。

**权衡：** 这些原则偏向谨慎而非速度。简单任务（翻译、格式化）凭直觉即可。

---

## 1. 定义意图再写

**不要假设模型知道上下文。不要隐藏歧义。先想清楚再下笔。**

- 写之前明确：输出给谁看？什么格式？多长？什么场景用？
- 如果需求有歧义，在提示词中指明方向——不要让模型猜
- 如果一个提示词需要 500 字才能说清，停下来：拆成多个或用示例代替
- 上下文工程 > 提示词工程：失败通常源于上下文混乱，而非措辞

## 2. 精简优先

**用最少的字传达完整意图。不推测、不冗余。**

- 目标长度 150-300 字。超过 300 字的提示词应该被质疑
- 重要指令放开头或结尾——"lost in the middle" 效应会让中间内容被忽略
- 不写"顺便..."——模型会认真执行你随口一提的东西
- 迭代精简：写最简短版本 → 看输出缺什么 → 只加针对性修复

## 3. 结构驱动

**用结构化方式组织提示词。每个段落有明确功能，一个提示词只做一件事。**

- 用分隔标记区分部分：Claude 用 XML 标签，GPT 用 Markdown 标题
- 正面指令 > 否定指令：错"不要写太长" → 对"控制在 200 字以内"
- 提供 3-5 个多样化示例（few-shot），覆盖边界情况
- 一个提示词只做一件事——"分析数据然后写报告顺便想标题"拆成三个

## 4. 输出导向

**把模糊指令转化为可验证的成功标准。像对待代码一样对待提示词。**

- 错"写个好提示词" → 对"写一个 200 字以内的提示词，包含角色、指令、约束三部分，用 XML 标签分隔"
- 测试优先：写最简草稿 → 用 2-3 个不同输入测试 → 只针对失败模式修复
- 多步骤任务每步独立验证：`步骤 → 验证: [检查点]`
- 维护基准测试集，每次修改提示词后回归验证

---

**这些原则在起作用的表现：** 同一提示词对不同输入产出一致质量的输出、模型严格遵守约束不多不少、第一次写的提示词就能用。
