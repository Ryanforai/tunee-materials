# Agent 产品专项审查清单

> **适用场景**：PRD 涉及多节点工作流、提示词、上下文传递、工具调用、模型选择中的**任意一项**。  
> **使用方法**：通用审查（5 个问题）过完后，Agent 类产品必须再过这 5 个区。每个区都有必查清单、PRD 要写的模板、和真实踩坑案例。

---

## A. 提示词变更

### 为什么重要

提示词看起来只是"几段文字"，但它其实是 Agent 的核心协议。改提示词常常伴随着**参数增减、输出结构变化、输出风格变化**——研发看成"文案微调"，联调时才炸。

### 必查清单

| 问题 | 在 PRD 里找什么 |
| --- | --- |
| 提示词有变化吗 | 明确说清楚：是调整表达、还是加新参数、还是改输出格式 |
| 输入参数变了吗 | 列出所有新增/删除/改名/改类型的参数 |
| 输出结构变了吗 | 列出字段名、类型、枚举值变化 |
| 哪些地方会用到 | 列出所有调用这个提示词的模块 / 节点 |
| 同步了谁、没同步谁 | 每个调用方明确标记已同步 / 待同步 |

### PRD 要写的模板（可直接复制）

```
【提示词变更说明】

- 变更类型：[表达调整 / 新增参数 / 删除参数 / 输出格式变更]
- 变更位置：<哪个节点的哪个提示词>

输入参数变化：
| 参数 | 变化 | 类型 | 来源 | 默认值 |
| --- | --- | --- | --- | --- |
| style_tag | 新增 | string | 用户选择 | "默认" |
| role | 删除 | - | - | - |

输出结构变化：
| 字段 | 原 | 现 | 影响 |
| --- | --- | --- | --- |
| scenes | List[string] | List[Object{text, duration}] | 前端解析逻辑要改 |

影响范围：
| 模块/节点 | 使用了哪个参数或字段 | 同步状态 | 负责人 |
| --- | --- | --- | --- |
| 前端 Chat 组件 | role | ✅ 已改为 user_type | 张三 |
| 服务端路由 | role | ⚠️ 未改，本次一起改 | 李四 |
| 下游 style_match 节点 | scenes | ⚠️ 需适配新格式 | 王五 |
```

### 典型场景（示例）

> MV Creative Director 某次调整提示词，把 understand 节点的 `role` 字段去掉，只通知了前端。服务端的路由逻辑也读这个字段，上线后路由错乱，回滚 1 次。

## B. 上下文管理

### 为什么重要

Agent 在节点之间传递的上下文很容易失控：**用了但没说用什么、丢了但没兜底、太长但没截断、截断了但影响效果**。这些问题 PRD 里经常完全不提，上线后才发现。

### 必查清单

| 问题 | 在 PRD 里找什么 |
| --- | --- |
| 上下文里有什么 | 用户输入、历史对话、系统状态、上游节点输出，分别列清楚 |
| 节点之间怎么传 | 传全量、摘要、还是只传 ID？ |
| Token 预算多少 | 预估 token 数，接近模型上限时如何处理 |
| 超长怎么截断 | 保留头部、尾部、还是摘要？截断了下游会不会报错 |
| 上下文丢失会怎样 | 重试时上下文还在吗？会话重开后如何恢复 |
| 跨节点一致性 | 同一个概念在不同节点用不同字段名吗（比如 user_id vs uid） |

### PRD 要写的模板

```
【上下文设计】

节点间传递内容：
| 节点 | 传入 | 传出 | Token 估算 |
| --- | --- | --- | --- |
| understand | 用户输入 + 历史 3 轮对话 | 意图分类 + 关键词 | ~500 |
| story_generation | 意图 + 关键词 + 歌词全文 | 场景列表 + mood | ~2000 |
| visions_generation | 场景列表 + mood + 风格 | 画面描述列表 | ~3000 |

超长处理：
- 单节点上下文 > 8000 token：保留最新 3 轮对话 + 当前请求 + 系统 prompt
- 歌词超长（>1000 字）：分段处理，每段 400 字，overlap 50 字
- 触发截断时：日志埋点 `context_truncated`，参数记录原长度

上下文丢失场景：
| 场景 | 处理 |
| --- | --- |
| 生成任务中断后重试 | 从最近一次检查点恢复（任务状态存 Redis，TTL 24h）|
| 用户关闭页面再打开 | 按 task_id 恢复进度和上下文 |
| 服务重启 | 持久化到 DB 的任务可恢复，内存态丢失 |
```

### 典型场景（示例）

> 某次音乐生成任务，用户歌词 1500 字，story 节点直接把全文塞给模型，超出 token 限制被静默截断，结果只生成前半段的场景，上线后用户投诉"后半段没音乐"。

## C. 工具调用

### 为什么重要

Agent 调用的每个工具都是一个"外部依赖"——工具 API 变了、超时了、限流了、返回格式变了，任何一个都可能让 Agent 崩。PRD 里光说"调用 xxx 工具"不够，必须说清楚**调什么、怎么调、挂了怎么办**。

### 必查清单

| 问题 | 在 PRD 里找什么 |
| --- | --- |
| 要调哪些工具 | 列清单：每个工具的名字、版本、用途 |
| 输入输出 schema | 每个工具的参数和返回值定义 |
| 调用时机 | 什么条件下调、能不能并行 |
| 失败怎么处理 | 超时、限流、鉴权失败、返回格式异常——每种都要有策略 |
| 重试策略 | 重试几次、间隔多久、重试是否消耗用户额度 |
| 降级方案 | 工具完全不可用时，Agent 是直接失败还是走备选路径 |
| 版本更新同步 | 工具 API 升级时，Agent 怎么跟上（硬编码版本是红灯）|
| 用户感知 | 用户能看到工具在跑吗？大概多久出结果？失败文案是什么 |
| 成本和配额 | 每次调用成本多少、月度预算、QPS 限制 |

### PRD 要写的模板

```
【工具调用清单】

| 工具 | 用途 | 输入 | 输出 | 预期耗时 | 失败处理 |
| --- | --- | --- | --- | --- | --- |
| Mureka V8 | 音乐生成 | 歌词 + 风格 | 音频 URL | 30-60s | 超时重试 1 次，仍失败提示"换个风格再试" |
| Seedance 2.0 | MV 画面生成 | 场景描述 + 风格 | 视频片段 | 60-120s | 单片段失败只重试该片段，整体成功率 > 80% 就出 |
| Lip-sync API | 口型对齐 | 视频 + 音频 | 对齐后视频 | 20-40s | 失败降级：直接出没对齐的版本 + 提示 |

通用失败策略：
- 超时阈值：单次工具调用不超过 120s，整任务不超过 5min
- 重试：只对可重试错误（超时、5xx、限流）重试，**最多 2 次**
- 重试不消耗用户额度
- 连续失败 3 次后：任务标记失败，日志记录原因，用户看到具体失败类型提示

版本管理：
- 所有工具的 endpoint、版本号走 `tool-config` 配置中心
- 禁止代码里硬编码 API URL 或版本号
- 工具升级时需经过：沙箱验证 → 灰度 5% → 全量
```

### 典型场景（示例）

> MV 生成里的 Seedance API 某次升级，返回结构从 `{url, duration}` 改为 `{clips: [{url, duration}]}`。代码里直接读 `url` 字段，上线后所有 MV 生成失败 12 小时才被发现。

## D. 节点数据流（上下游影响矩阵）

### 为什么重要

Agent 是多节点编排出来的。改一个节点，**上游要知道你改了**（可能要改输出）、**下游要知道你改了**（可能要改解析）、**旁路节点也可能受影响**（共享上下文或状态）。

PRD 里光说"改动节点 X"没用，必须给一张**影响矩阵**。

### 必查清单

| 问题 | 在 PRD 里找什么 |
| --- | --- |
| 改动点是哪个节点 | 精确定位到节点名 |
| 上游有谁 | 列出所有输出到这个节点的节点 |
| 下游有谁 | 列出所有消费这个节点输出的节点 |
| 旁路节点 | 共享上下文/状态的节点（比如都读同一个 session） |
| 字段级影响 | 新增/修改/删除字段分别影响哪些节点 |
| 兼容性策略 | 同时支持新旧格式？还是一刀切？过渡期多久 |
| 回滚方案 | 出问题怎么回滚、能不能单独回滚这个节点 |

### PRD 要写的模板

```
【改动点】
节点：story_generation  
变更：输出 scenes 字段从 List[string] 改为 List[Object]

【影响矩阵】

上游影响（输入给 story_generation 的节点）：
| 节点 | 输出字段 | 是否要调整 | 说明 |
| --- | --- | --- | --- |
| understand | intent, keywords | ❌ 不动 | 输出给 story 的参数不变 |
| lyric_parse | lyrics_segments | ❌ 不动 | - |

下游影响（消费 story_generation 输出的节点）：
| 节点 | 消费字段 | 影响 | 改法 | 负责人 | 完成日期 |
| --- | --- | --- | --- | --- | --- |
| visions_generation | scenes | ⚠️ 需适配 Object 结构 | 按新格式取 text 和 duration | 王五 | 04-28 |
| script_assembly | scenes | ⚠️ 需适配 | 同上 | 王五 | 04-28 |
| lip_sync_prep | - | ❌ 不读 scenes | 无 | - | - |

旁路影响（共享上下文的节点）：
| 节点 | 共享的状态 | 是否受影响 | 说明 |
| --- | --- | --- | --- |
| monitoring | task_progress | ❌ 不读 scenes 字段 | - |
| billing | task_meta | ❌ 不读 scenes 字段 | - |

【兼容性】
- 过渡期：新旧格式并存 2 周
- 所有下游节点需在 04-28 前完成适配
- 04-28 后 story_generation 只输出新格式

【回滚】
- 单节点回滚可行：story_generation 回到旧版本即可，下游可同时支持新旧格式
```

### 典型场景（示例）

> 之前某次改 understand 节点的输出去掉 role 字段，只同步了前端。服务端的路由、下游 style_match 都读这个字段，当天回滚。教训：**凡是改节点输出，必须出影响矩阵**。

## E. 模型选择

### 为什么重要

Agent 产品的能力上限就是模型的能力上限。选错了模型、没验证效果、没 Fallback 方案，整个功能就是纸上谈兵。

### 必查清单

| 问题 | 在 PRD 里找什么 |
| --- | --- |
| 选了什么模型 | 明确到模型 + 版本（比如 Mureka V8、不是"音乐模型"）|
| 为什么选这个 | 至少 2 个候选模型的对比：效果 / 成本 / 延迟 |
| 效果验证 | Demo、截图、对比样本，不能只写"理论上可行"|
| 成本预估 | 单次调用成本、预期 QPS、月度预算 |
| 延迟可接受性 | P50 / P95，用户能否接受 |
| 失败率 | 在不同输入下的稳定性 |
| Fallback 策略 | 主模型挂了走哪个、降级后用户怎么通知 |
| 模型切换机制 | 将来想换模型时，代码改动多大（能不能配置化）|
| 输出格式稳定性 | 模型输出格式不稳定时怎么处理（JSON 解析失败等）|

### PRD 要写的模板

```
【模型选择】

| 维度 | 主选：Mureka V8 | 候选 1：Tempolor 4.5 | 候选 2：MiniMax 2.5 |
| --- | --- | --- | --- |
| 效果 | 人声 9/10, 编曲 8/10 | 人声 7/10, 编曲 8/10 | 人声 8/10, 编曲 7/10 |
| 成本 | $0.08/首 | $0.05/首 | $0.06/首 |
| 延迟 P50 | 45s | 30s | 50s |
| 延迟 P95 | 90s | 60s | 120s |
| 稳定性 | 98% 成功 | 95% 成功 | 92% 成功 |
| 适用场景 | 付费用户主打 | 免费用户、快速试听 | 备选 |

【验证样本】
- 测试集：20 首各风格的对比（链接：<文档 URL>）
- 主观评分：Mureka > MiniMax > Tempolor（3 人盲评）

【Fallback 策略】
- 主：Mureka V8
- 备 1：Tempolor 4.5（Mureka 超时或连续失败）
- 备 2：MiniMax 2.5（前两个都不可用时）
- 降级时用户提示："正在为你切换生成引擎，预计多等 30 秒"

【输出稳定性】
- Mureka 返回格式偶尔会多返回字段：业务层按 schema 校验，多余字段忽略
- JSON 解析失败：重试 1 次，仍失败则视为模型调用失败

【模型切换机制】
- 所有模型 ID、endpoint 走 `model-config`
- 运营后台可调整：主模型、Fallback 顺序、灰度比例
- 禁止代码里写死模型 ID 或版本号
```

### 典型场景（示例）

> 早期版本直接在代码里写 `model = "mureka-v7"`，Mureka 升级到 V8 后 API 路径变了，线上报错一整晚。教训：**模型版本必须配置化，不硬编码**。

## 审查时的用法

过 Agent 类 PRD 时，按 A → B → C → D → E 顺序逐区检查。**每个区都要在 PRD 里找到明确答案**——模糊或缺失就记到审查报告里。

### 优先级判断

- **🔴 必须改**：D 区没有影响矩阵 / E 区没有效果验证 / C 区工具失败没策略。这些是**100% 会返工**的。
- **🟡 强烈建议改**：A/B/C 区信息不全。评审会上基本会被追问，早补早省事。
- **🟢 细节优化**：已经说清楚但可以写得更结构化的部分。

### 写建议时的要求

每一条建议都要做到：
1. **具体**：点名到字段、节点、接口——不是"考虑一下 xxx"
2. **可执行**：给出直接可用的模板、表格——不是"建议补充"
3. **有影响**：说清楚不改会在哪里爆——帮助产品评估优先级
