# PRD 审查

## 这是做什么的

评审会前的自检工具。用 10-15 分钟过一遍 PRD，提前发现：
- 评审会上会被追问、卡住的地方
- 上线后会踩坑、返工的地方
- 跨团队协作时会掉链子的地方

**目标读者**：产品经理（写完 PRD 自检）、tech lead（评审前预审）、新人（学习怎么写好 PRD）。

---

## 审查流程

### 第一步 · 读懂 PRD

把 PRD 读进来（文本、.md / .docx / .pdf 文件、飞书 / Notion 链接都行），然后**用一句话复述**：

> 这个需求做什么、给谁用、解决什么问题、最终交付物是什么。

如果一句话说不清楚，PRD 的"背景目标"部分肯定有问题。

**注意**：如果用户只给了几句话草稿，不要硬审。直接告诉他这还不够可审文档，列出最基础的缺失（通常就是"为什么做、做什么、给谁"都没交代），让他补到一定程度再来。

### 第二步 · 通用审查（5 个问题）

详细清单在 `checklist.md`。这 5 个问题每份 PRD 都要过：

| 问题 | 重点 |
| --- | --- |
| 1. 为什么做？ | 具体问题 + 证据 + 成功指标（可测量） |
| 2. 具体做什么？ | In/Out of scope + 和现有功能的关系 |
| 3. 正常怎么走？ | 主流程、关键字段、结束状态 |
| 4. 不正常怎么办？ | 异常、边界、权限、配额、计费 |
| 5. 怎么知道成了？ | 指标定义（分子/分母/基线/目标值）+ 埋点 |

### 第三步 · Agent 专项审查（5 个区）⭐

**这一步是重点**。详细在 `checklist-agent.md`。Agent 类产品的返工几乎都出在这 5 个地方：

| 区 | 检查什么 |
| --- | --- |
| A. 提示词变更 | 提示词改了？输入输出结构变了？谁同步了、谁没同步？ |
| B. 上下文管理 | 上下文里放了什么？Token 够吗？节点间怎么传？丢了会怎样？ |
| C. 工具调用 | 调哪些工具？失败怎么办？版本怎么管？用户感知到了吗？ |
| D. 节点数据流 | 改了一个节点，上下游都跟上了吗？字段变更有影响矩阵吗？ |
| E. 模型选择 | 为什么选这个模型？效果/成本/延迟验证了吗？Fallback 有吗？ |

**只要 PRD 涉及多节点、提示词、工具调用、模型切换，这 5 个区必须逐个过。**

### 第四步 · 常见坑对照

看 `checklist-gotchas.md`，贴合 Tunee 的 6 个高频事故，逐条对照 PRD。命中一个就记一个，每条都配可复制的 PRD 改法。

### 第五步 · 输出报告

按下面的模板。每条发现必须有 4 段（**问题在哪 · 为什么要改 · 改成什么样 · 影响范围**），少一段等于没审。

---

## 报告模板

```markdown
# PRD 审查：<需求名称>

## 一眼概况
- **理解**：<一句话复述：做什么、给谁、解决什么>
- **能否进评审**：✅ 可以 / ⚠️ 改改再评 / ❌ 得重写
- **最大风险**：<最可能返工的一个点>
- **涉及的 Agent 区域**：A 提示词 / B 上下文 / C 工具 / D 数据流 / E 模型（勾选适用的）

## ⭐⭐⭐ 必须改（N 条，评审前搞定）

### 1. <一句话标题，要能独立看懂>
- **问题**：<具体缺了什么，点名字段/接口/用户分层/节点>
- **为什么要改**：<不改会在哪个环节爆、谁返工、上线后什么症状>
- **改成**：
  > <可以直接 ctrl+c ctrl+v 进 PRD 的一段文字或表格>

### 2. ...

## ⭐⭐ 强烈建议改（N 条，评审会上会被追问）

...

## 🟢 细节优化（N 条，选改）

...
```

## 写法要点

### ❌ 禁止这样写
- "边界情况考虑不充分"（太虚）
- "需要补充埋点"（哪个埋点）
- "要考虑前后端影响"（哪一侧）
- "建议完善异常路径说明"（完善成什么样）

### ✅ 要这样写

**具体到字段 / 接口 / 节点**：

> ❌ "边界情况不充分"  
> ✅ "用户免费额度刚好用完的瞬间点'生成'——PRD 没说是弹付费墙还是直接报错；未登录游客态能否看到入口也没说"

**具体到影响谁**：

> ❌ "前后端要注意"  
> ✅ "3.2 节改了 understand 节点输出的 `role` 字段，服务端路由也读这个字段，但 PRD 只提了前端；下游 style_match 节点的输入也依赖 `role`，需同步"

**"建议补充"要给成品**：

> ❌ "建议补充异常处理说明"  
> ✅  
> **异常处理补充：**
> - 接口超时（>10s）：前端展示"生成超时，是否重试？"，重试**不消耗**额度；连续 2 次失败后引导反馈
> - 违禁内容：返回具体违规类型（涉政/涉黄/版权），引导用户修改而非直接拒绝
> - 额度用完：免费用户弹付费墙（路径 A），付费用户展示余额与续费入口（路径 B）

## 边界情况

| 情况 | 怎么处理 |
| --- | --- |
| PRD 只有几句话草稿 | 不硬审。告诉用户这还不够可审文档，列出最基础缺失 |
| PRD 提到图但图没给 | 明确指出第几节提到图但图没给，不要凭空假设图里有什么 |
| 用户只想审某部分 | 尊重范围，但结尾提示"本次只审了 X，完整评审建议再过 Y、Z" |
| PRD 是英文 | 报告跟随 PRD 语言：中文 PRD 出中文报告，英文 PRD 出英文报告 |
| 内部不可见的改动（提示词、配置） | **不降低要求**。这类改动的横切影响往往更大 |
| PRD 写得很好 | 不凑数字硬挑 🔴。七个区域都合格就说"可以进评审"，只列 🟢 建议 |
| PRD 写得很烂 | 识别**真正的阻塞项**（≤ 5 条）。太多就直接建议整体重写，不要列 20 条🔴吓跑产品 |

---

## 最后

报告"一眼概况"的"能否进评审"要给**明确结论**，不含糊。

产品看完报告之后应该**知道现在改什么、改完就能开评审会**——而不是拿到一份 punchlist 不知从何下手。
