# PRD 高频踩坑清单

> **用法**：审 PRD 时逐条对照。每一条都对应过 AI 产品的真实返工或线上事故。命中一条就在报告里用"PRD 里该怎么写"的格式给出具体建议。  
> **说明**：这份是**通用高频坑**。Agent 特有的提示词、上下文、工具调用、节点数据流、模型选择的坑在 `checklist-agent.md`，不重复。

---

## 坑 1：生成任务的时间和状态没说清

### 典型场景

音乐 / MV / AI 视频生成类需求，PRD 只写"用户点击生成→返回结果"，没说：
- 要等多久（用户会不会以为卡死了）
- 等待过程中能不能退出
- 失败分哪几类，分别提示什么
- 同时能发几个任务

**结果**：上线后用户投诉"一直转圈""失败了不知道为什么""退出一次全没了"。

### PRD 要写的模板

```
【生成流程说明】

预期耗时：
- 简单生成：30-60 秒（P50 45s）
- 复杂生成（多段落 MV）：60-180 秒（P95 150s）

用户交互：
- 生成过程中可切后台，任务继续在云端跑
- 可主动"取消"：取消后不扣费，但已生成部分丢失
- 完成后推送提醒（App 内 + 可选推送）

失败分类：
| 失败类型 | 用户看到的提示 | 可否重试 | 是否扣费 |
| --- | --- | --- | --- |
| 内容违规 | "歌词含违规内容，请修改关键词：xxx" | 修改后可 | 否 |
| 服务超时 | "生成超时，系统自动重试中" | 自动重试 1 次 | 否 |
| 服务故障 | "服务繁忙，可稍后再试" | 用户手动 | 否 |
| 额度不足 | "本月免费次数已用完，升级会员继续创作" | 升级后 | - |
| 风格歌词严重不匹配 | "当前歌词不太适合 xxx 风格，建议换成 yyy" | 可 | 否 |

并发限制：
- 免费用户：单人同时 1 个任务
- 付费用户：单人同时 3 个任务，超出进入排队
- 排队显示位置："前方还有 X 个任务，预计等待 Y 分钟"
```

## 坑 2：计费规则模糊

### 典型场景

"按次扣费"听起来简单，实际上至少有 10 种场景都要说清楚：成功扣、失败不扣、重试不扣、取消不扣、违规不扣、不同套餐不同价格、灰度期免费、海外汇率差异⋯⋯

**结果**：客服被问爆，用户投诉"为什么失败了还扣费"。

### PRD 要写的模板

```
【计费规则】

| 场景 | 是否扣费 | 金额 | 备注 |
| --- | --- | --- | --- |
| 生成成功 | 是 | 1 次 | 按订阅套餐定义 |
| 主动取消（< 5s 内）| 否 | 0 | 防止误点 |
| 主动取消（> 5s）| 是 | 0.5 次 | 已产生算力成本 |
| 超时自动重试 | 否 | 0 | 系统原因不计费 |
| 内容违规 | 否 | 0 | 引导用户修改 |
| 额度不足拒绝 | 否 | 0 | 直接拦截 |
| 付费用户 A/B 实验组 | 特殊 | 实验期免费 | 实验结束后按规则 |

不同用户分层：
| 分层 | 月度额度 | 单次成本 |
| --- | --- | --- |
| 免费用户 | 3 次 | - |
| 月度会员 | 50 次 | 超出 $0.99 |
| 年度会员 | 100 次 | 超出 $0.79 |
| 企业用户 | 按合同 | 定制 |

灰度期特殊处理：
- 灰度期（上线第 1 周）所有用户免费
- 灰度期使用次数不计入月度额度
- 客服话术需同步：<链接>

退款规则：
- 生成内容违规导致的用户重做：返还次数
- 用户主观不满意：不退（引导重试）
- 系统故障导致的异常扣费：自动返还 + 站内信通知
```

## 坑 3：和现有活动 / 弹窗的冲突没说

### 典型场景

新加一个"风格推荐"弹窗，但没说和现有的"活动公告""用户反馈""新功能引导"弹窗的优先级和频次规则。

**结果**：上线后用户打开 App，四五个弹窗同时弹，体验崩坏。

### PRD 要写的模板

```
【弹窗协调】

与现有弹窗的关系：
| 现有弹窗 | 优先级 | 和本需求弹窗的关系 |
| --- | --- | --- |
| 活动公告（运营）| P0 | 本需求等它关闭后才弹 |
| 新版本更新提示 | P0 | 同上 |
| 风格推荐（本需求）| P1 | - |
| 用户反馈邀请 | P2 | 在本需求之后弹 |
| 上月数据报告 | P3 | 最后弹 |

触发频次：
- 单用户每天最多看 1 次
- 点击关闭后 7 天不再弹
- 点击"下次再说"后 24 小时不弹
- 点击"不再显示"后永久不弹

触发条件：
- 用户首次进入生成页
- 已完成 >= 3 次生成
- 过去 7 天未看过本弹窗
```

## 坑 4：指标定义不可计算

### 典型场景

PRD 写"提升完成率 20%"，但没定义"完成率"的分子分母。等上线后做数据分析时，发现：
- 完成率到底是"提交 → 成功" 还是"打开页面 → 成功"？
- task_id 怎么去重？
- 时间窗是日级还是周级？
- 基线值是多少？

**结果**：上线后无法判断成功，团队扯皮。

### PRD 要写的模板

```
【成功指标】

北极星指标：MV 生成完成率 ≥ 75%

定义：
- **分子**：事件 `mv_generate_success`（按 task_id 去重）
- **分母**：事件 `mv_generate_submit`（按 task_id 去重）
- **时间窗**：日级 + 7日滚动
- **基线值**：当前 60%（2026-04 数据，链接：<报表 URL>）
- **目标值**：75%（上线后 4 周内达成）
- **数据源**：数据平台 > MV 生成漏斗

辅助指标：
| 指标 | 定义 | 基线 | 目标 |
| --- | --- | --- | --- |
| 首次生成成功率 | 新用户首次提交成功率 | 55% | 70% |
| 平均重试次数 | 每任务平均 submit 次数 | 1.8 | 1.3 |
| 失败原因 TOP 3 | failure_reason 分布 | 违规/超时/额度 | 明确归因 |

分层看数据：
- 按用户类型（免费 / 付费 / 企业）
- 按风格类别（云汐 / K-pop / ⋯）
- 按地区（T1 / T2 / T3）
- 按设备（Web / iOS / Android）
```

## 坑 5：埋点漏了分层字段

### 典型场景

核心事件埋了，但没带用户分层、地区、设备这些字段。上线后想看"付费用户的完成率是不是更高"，发现事件里没有 user_type 字段。

**结果**：要么等一个版本加埋点，要么从 join 其他表——都是成本。

### PRD 要写的模板

```
【埋点规范】

每个关键事件必带的基础参数：
| 参数 | 类型 | 取值说明 |
| --- | --- | --- |
| user_type | enum | free / monthly / annual / enterprise |
| user_age_days | int | 用户注册天数（新老用户区分）|
| region | string | T1 / T2 / T3 / custom |
| device | enum | web / ios / android |
| app_version | string | - |
| session_id | string | 会话级去重用 |

新增核心事件：
| 事件名 | 触发时机 | 关键参数 |
| --- | --- | --- |
| mv_generate_submit | 用户点"生成"按钮 | task_id, style, lyric_length, prompt_version |
| mv_generate_success | 生成完成 | task_id, duration_ms, model_used |
| mv_generate_fail | 生成失败 | task_id, failure_reason, retry_count |
| mv_generate_cancel | 用户主动取消 | task_id, elapsed_ms, cancel_reason |

修改的事件：
- `mv_preview_click`：参数 `style_id` 改名为 `style_category`，旧字段保留 2 周
```

## 坑 6：依赖的 AI 能力没验证就进评审

### 典型场景

PRD 写「用 Seedance 2.0 做 MV、用 Mureka 做音乐」——但没有任何 demo、截图、效果对比、成本估算、延迟数据。评审会讨论 1 小时，研发要求先 POC，整个需求延期一个版本。

**结果**：评审白开，团队时间浪费。

### 审查时的标准

**对于依赖第三方 AI 能力 / 模型的需求，以下任意一项缺失，直接标 🔴 必须改：**

- Demo / POC 结果（至少 3 个有代表性的样本）
- 成本估算（单次成本 + 预期 QPS + 月度预算）
- 延迟结论（P50 / P95）
- 在边界 case 下的表现（超长输入、小众风格、多语言等）
- 失败率（< 5% 才算稳定）

### PRD 要写的模板

```
【AI 能力验证】

能力：Seedance 2.0 视频生成

验证样本：
- 测试集：20 个不同风格的 prompt（链接：<内部文档>）
- 输出样本：[链接到 3 段代表性视频]
- 对比模型：Seedance 1.5 vs Sora 2.0 vs MiniMax Video（<对比文档>）
- 结论：在 XX 风格上 Seedance 2.0 明显优于其他两个

成本：
- 单次生成成本：约 $0.12（5 秒视频）
- 预期 QPS：日均 500，高峰 50
- 月度预算：约 $2,000

延迟：
- P50：60s
- P95：120s
- 最大超时阈值：180s（超过判定失败）

稳定性：
- 最近 2 周测试：成功率 94%
- 主要失败原因：超长 prompt（8%）、特殊风格（4%）
- 失败后的 fallback：见 checklist-agent.md E 区

已知边界：
- Prompt 中文长度 > 200 字时效果下降
- 某些风格组合（如"赛博朋克 + 水墨"）成功率低
- 人物正面特写容易出现崩脸，需配合 face-fix
```

## 用法小结

审 PRD 时**快速过这 6 个坑**，命中哪个就记哪个。**每条建议都要给可复制的模板，不能只说"建议补充"**。

**优先级参考**：
- 坑 2（计费）、坑 6（AI 能力验证） 漏了一般是 🔴 必须改
- 坑 1（生成流程）、坑 4（指标定义） 漏了是 🟡 强烈建议改
- 坑 3（弹窗）、坑 5（埋点字段） 漏了是 🟡 或 🟢 视重要性

**Agent 类产品**还要再过 `checklist-agent.md` 的 5 个专项区域。
