# MV Agent 原子能力 — 交叉验证报告

> 日期：2026-04-26
> 方法：4 个子 Agent 独立调研后交叉验证，与现有设计稿对比

---

## 调研维度

| Agent | 调研角度 | 核心发现 |

## 一、新增原子能力清单

### L0: UI 模式层（新增）

| # | 能力名 | 方向 | 一句话 | 优先级 |
|---|--------|------|--------|--------|
| **M1** | `activate_workspace` | Agent → UI | 检测 MV 意图 → 全屏对话变为 30% 对话 + 70% 画布 | P0 |
| **M2** | `deactivate_workspace` | Agent → UI | 任务完成/用户闲聊 → 收起画布，回到全屏对话 | P1 |
| **M3** | `bind_tools_to_mode` | 系统 | 根据当前 UI 模式切换可用工具集（画布模式下绑定画布工具） | P1 |

**M1 详述**：这是用户提到的"识别到要做 MV 再打开画布"的能力。当前文档没有这个原子能力。
- 触发时机：Agent 识别到 mv_creation / generate_image / generate_video 等意图时
- UI 行为：全屏 chat → split layout (360px chat + flex canvas)
- 同时初始化：创建第一个 Page，设置默认 region 布局

**M2 详述**：IntentWeave 论文提到的"Surface Retreat"概念。任务完成后不应让用户卡在重型界面里。
- 触发时机：MV 导出完成 / 用户长时间无操作 / 用户说"好的谢谢"
- UI 行为：split → fullscreen chat，画布保留但不占据视口

## 二、基础设施能力（非原子能力，但必须在架构层支持）

以下能力不属于任何 Agent Tool，而是系统基础设施：

| # | 能力 | 说明 |
|---|------|------|
| **I1** | Shared State Protocol | 画布和 Agent 共享单一事实源（CRDT 或等价方案），防止 Agent 覆盖用户修改 |
| **I2** | Event Ordering | 每个画布事件和 Agent 命令携带 Lamport 时间戳，检测并发冲突 |
| **I3** | Asset Dependency Graph | 追踪资产之间的依赖关系（Shot_5 依赖 character_ref_v3），修改上游时通知下游 |
| **I4** | Asset Lifecycle State | 资产状态机：draft → in-review → approved → deprecated → archived |
| **I5** | Asset Provenance | 每个资产记录完整生成链：模型、prompt、输入资产、生成时间 |

## 六、关键洞察

1. **最大的遗漏是 `activate_workspace`**：这是整个产品的"入口门"。没有它，对话和画布之间的切换就没有触发机制。这个能力应该是 Agent 识别到 MV 相关意图后第一个调用的。

2. **角色一致性（T10）和帧连续性（T8）应该从 LLM 内部能力提升为独立 Tool**：现有文档把它们归类为"Agent 内部逻辑"，但实际上需要确定性阈值和可量化的评分，不能靠 LLM 主观判断。

3. **内容安全（P7）是法律风险**：生成内容可能包含真实品牌 logo、名人面部、版权素材。必须在推送到画布之前自动检测。

4. **快照（C15/C16）比想象中重要**：用户在画布上做了大量手动调整后，需要一个"安全点"。这不是 undo（撤销单步操作），而是恢复到某个已知的良好状态。

5. **Ghost Preview（C12）是一种新的交互范式**：与其直接修改画布，不如先展示"如果这样改会是什么样"，让用户决定。这降低了 Agent 犯错的成本。
