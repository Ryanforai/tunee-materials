# MV Agent 原子能力体系 — 设计文档

> 日期：2026-04-26 | 状态：已确认
> 背景：从"剪辑台"升级到"对话式 Agent + 无限画布"架构

---

## 一、我们先厘清一个核心问题

### 什么是"原子能力"？

不是所有功能都应该叫原子能力。需要一个清晰的判定标准。

**判定原则：原子能力 = Agent 做不了、必须交给外部的事**

| 判断维度 | 是原子能力 | 不是原子能力 |
|---------|-----------|-------------|
| 需要计算像素/生成文件 | generate_image, ffmpeg | 文字总结（LLM 本身能做） |
| 需要访问外部存储 | get_asset, upload_asset | 读 State（内存数据） |
| 需要调用外部 API | video_model_api | 选择哪个模型（查表推理） |
| 需要操作前端 DOM | push_to_canvas | 生成回复文字（LLM 本身能做） |
| 需要确定性执行 | 任何 tool | 意图识别（LLM 推理） |

**关键结论：LLM 本身就是"大脑"，它的推理、文字生成、规划能力不需要注册为工具。原子能力是大脑的"手脚"。**

## 二、从用户的视角看：用户会做什么？

在设计原子能力之前，先穷举用户在 MV Agent 里可能做的事。这决定了我们需要哪些原子能力。

### 用户行为全景（非穷举，按意图分类）

```
用户进入对话
  │
  ├── "帮我做个 MV"          → 从 0 到 1 的完整流程
  ├── "为这首歌做个 MV"       → 上传音频 → 分析 → 全流程
  ├── "生成一张赛博朋克主角图" → 单点生成，不走路
  ├── "搜索王家卫的 MV 参考"   → 找灵感
  ├── "方案 B 不错，继续"      → 流程推进
  ├── "Shot 3 的脸歪了，重做"  → 反馈修改
  ├── "质量怎么样？"           → 状态查询
  ├── "帮我把这几个镜头拼接起来" → 编辑操作
  ├── "换成横屏 16:9"         → 格式转换
  ├── "提取这首歌的 BPM"       → 音频分析
  ├── "把字幕烧进去"           → ffmpeg 操作
  ├── "导出成片"               → 最终输出
  └── "你好"                   → 闲聊
```

**归纳为 6 类意图**（已有文档定义，这里确认其完整性）：

| 意图类 | 特征 | 需要的原子能力集 |
|--------|------|----------------|
| 自由操作 | 零前置，用户明确要什么 | 单个 or 少量工具 |
| 流程推进 | "继续"、"下一步" | 取决于当前阶段 |
| 链式执行 | 一句话触发多步 | 多个工具编排 |
| 反馈修改 | "换个风格"、"脸歪了" | 定位目标 + 重新生成 |
| 状态查询 | "到第几步了" | 读 State，无工具 |
| 闲聊 | 无关话题 | 纯 LLM 回复 |

## 三、原子能力分层模型

经过调研和交叉验证，我认为应该分 **五层**，而不是一锅端：

```
┌───────────────────────────────────────────┐
│  Layer 0: UI Mode（界面模式层）             │
│  控制全屏对话 ↔ 对话+画布的切换             │
│  例：activate_workspace, deactivate_...    │
├───────────────────────────────────────────┤
│  Layer 4: Skill（技能组合层）               │
│  编排多个 Tool + LLM 推理，完成完整子任务    │
│  例：generate_proposals, mv_planner         │
├───────────────────────────────────────────┤
│  Layer 3: Tool（外部工具层）                │
│  需要调用外部系统的事                        │
│  例：generate_image, run_ffmpeg             │
├───────────────────────────────────────────┤
│  Layer 2: Canvas Action（画布操作层）        │
│  影响前端 UI 的操作                          │
│  例：push_to_canvas, open_page, focus_page  │
├───────────────────────────────────────────┤
│  Layer 1: Perception（感知层）              │
│  读取/理解用户上传的内容                     │
│  例：read_asset, analyze_audio, parse_lyrics│
└───────────────────────────────────────────┘
```

**为什么分四层？** 因为"生成图片"和"把图片放到画布上"是完全不同的事：
- 生成图片 = 调用外部 API（L3）
- 放到画布 = 操作前端 DOM（L2）
- 读用户上传的图 = 感知内容（L1）
- 生成角色图 + 场景图 + 排版 = 技能组合（L4）

## 四、各层原子能力详细定义

### Layer 0: UI Mode（界面模式层）— "切"

这些能力控制 **全屏对话 ↔ 30% 对话 + 70% 画布** 的切换。是 Agent 识别意图后第一个触发的能力。

| # | 能力名 | 方向 | 输入 | 输出 | 说明 | 优先级 |
|---|--------|------|------|------|------|--------|
| **M1** | `activate_workspace` | Agent → UI | intent_type, initial_layout? | { workspace_id } | 检测到 MV 意图 → 全屏对话变为 30% 对话 + 70% 画布，初始化第一个 Page 和默认 region 布局 | P0 |

**M1 详述**：这是整个产品的"入口门"。没有这个能力，用户说完"帮我做个 MV"后仍然停留在纯聊天界面。

- **触发时机**：Agent 识别到 mv_creation / generate_image / generate_video / analyze_music 等意图时
- **UI 行为**：`position: fixed` 的全屏 chat → `grid` 布局（360px chat + flex canvas）
- **副作用**：创建第一个 Page，设置默认 region 布局（music_area, proposal_area, visual_assets, timeline）
- **State 更新**：`workspace.active = true`, `workspace.current_page = page_1`

> 后续可补充 `deactivate_workspace`（任务完成/闲聊时收起画布，回到全屏对话），但 V1 不做。

### Layer 1: Perception（感知层）— "读"

这些能力的共同点：**输入是用户上传的内容，输出是结构化理解**。LLM 无法直接"看到"或"听到"，需要预处理。

| # | 能力名 | 输入 | 输出 | 底层实现 | 说明 |
|---|--------|------|------|---------|------|
| **P1** | `read_asset` | asset_id / URL | { url, type, size, metadata } | 存储 API | 读取用户上传的素材基本信息 |
| **P2** | `analyze_audio` | audio_url | { bpm, key, duration, mood_curve, structure, energy_curve } | 音频分析 API | 分析音乐，提取节拍、调性、情绪曲线 |
| **P3** | `parse_lyrics` | lyrics_text / lrc_url | { language, lines: [{time, text}], structure } | 文本解析 + LLM | 解析歌词，对齐时间戳 |
| **P4** | `analyze_image` | image_url | { description, colors, composition, faces[], style_tags } | VLM API | 理解图片内容，用于"帮我找类似风格的参考"等场景 |
| **P5** | `analyze_video` | video_url | { duration, fps, resolution, scenes[], motion_level } | VLM API + ffprobe | 理解用户上传的参考视频 |
| **P6** | `read_text_file` | text_url / text_content | { content, encoding, language } | 文件读取 | 读取用户上传的文本（剧本、分镜描述等） |

**讨论点 1**：P2 和 P3 是否可以合并为一个 `analyze_music`？
- 合并的好处：一次调用，统一返回
- 拆分的好处：用户可能只上传纯音频（无歌词），或只上传歌词文本（无音频）
- **我的建议**：合并为 `analyze_music`，但参数可选 `audio_url` 和 `lyrics_text`，至少提供一个

### Layer 2: Canvas Action（画布操作层）— "放"

这些能力**不产生新内容，只决定内容在哪里、以什么形式展示**。是前端操作，但由 Agent 触发。

| # | 能力名 | 输入 | 输出 | 底层实现 | 说明 |
|---|--------|------|------|---------|------|
| **C1** | `push_to_canvas` | event_type, target_region, data | { success } | 前端 EventBus | 将数据推送到画布的指定区域 |
| **C2** | `open_page` | page_id, page_title, initial_content? | { page_id } | 前端路由 | 在画布上新开一个 Page（类似浏览器新标签） |
| **C3** | `switch_page` | page_id | { success } | 前端路由 | 切换到指定 Page |
| **C4** | `close_page` | page_id | { success } | 前端路由 | 关闭一个 Page |
| **C5** | `list_pages` | — | [{ page_id, title, thumbnail, status }] | 前端状态 | 列出所有已打开的 Page |
| **C6** | `focus_element` | element_id | { success } | 前端定位 | 聚焦到画布上的某个元素（缩放+居中） |
| **C7** | `update_element` | element_id, patch_data | { success } | 前端更新 | 更新画布上已有元素的部分属性 |
| **C8** | `remove_element` | element_id | { success } | 前端删除 | 从画布移除元素 |
| **C9** | `layout_elements` | region, layout_type, elements | { success } | 前端排版 | 在指定区域内按某种布局排列元素 |

**讨论点 2**：`push_to_canvas` 和 `layout_elements` 是否需要分开？
- `push_to_canvas` 是"推数据"，告诉画布某个区域有了新数据
- `layout_elements` 是"调排版"，决定元素在画布上的排列方式（网格、堆叠、时间线等）
- **我的建议**：分开。前者是数据同步，后者是视觉编排。Agent 可能只想更新数据而不改布局。

**讨论点 3**：Page 的管理是否需要 Agent 控制？
- 用户说"先做 A 歌的 MV，再做个 B 歌的" → 需要开新 Page
- 但 Page 的开关也可以由用户手动完成（画布是用户的）
- **我的建议**：Agent 有 `open_page` 能力，但默认不主动开，除非用户明确要求。`switch_page` 和 `close_page` 同理。

### Layer 3: Tool（外部工具层）— "做"

这些是**真正产生新内容**的能力，需要调用外部 API 或系统。

| # | 能力名 | 输入 | 输出 | 底层实现 | 预计耗时 | 说明 |
|---|--------|------|------|---------|---------|------|
| **T1** | `generate_image` | prompt, asset_type, model?, seed?, size? | { image_url, seed, metadata } | 图片生成 API | 5-20s | 生成图片（角色/场景/分镜/参考） |
| **T2** | `generate_video` | video_prompt, frame_url?, duration, model? | { video_url / task_id, status } | 视频生成 API | 60-120s | 生成视频片段（异步任务） |
| **T3** | `run_ffmpeg` | input_urls, operation, params | { output_url, metadata } | ffmpeg | 1-30s | 音频/视频处理（拼接、提取、转码、烧字幕） |
| **T4** | `evaluate_quality` | asset_url, asset_type, criteria? | { score, issues[], suggestion } | VLM API | 3-10s | 质量检测 |
| **T5** | `web_search` | query, source?, max_results? | { references[] } | 搜索 API | 2-8s | 搜索参考素材 |
| **T6** | `lip_sync` | video_url, audio_url | { video_url (synced), metadata } | Lip-Sync API | 10-60s | 口型同步（歌词驱动 MV 场景） |
| **T7** | `face_swap` | source_video_url, face_image_url | { video_url (swapped) } | Face-Swap API | 10-30s | 角色一致性跨镜头 |

**讨论点 4**：T6 和 T7 是必须的，还是可以合并到 T2 中？
- Kling 3.0 的 infinite_talk 模式已经内置 lip-sync
- Seedance 2.0 有 lipsync 模式
- **我的建议**：独立为原子能力。因为模型能力在变化，今天内置的明天可能不内置。独立成 tool 可以随时切换底层实现。

**讨论点 5**：是否需要 `generate_audio`（TTS / 音效生成）？
- MV 场景中用户通常上传已有音频
- 但如果用户想做"先有 MV 再生成配乐"，就需要这个能力
- **我的建议**：V1 不做。但保留扩展位，参数设计时预留。

### Layer 4: Skill（技能组合层）— "编"

Skill = LLM 推理 + 多个 Tool 的编排。是"半成品工作流"。

#### 编排型 Skill（6 个）

| # | Skill 名 | 触发条件 | 编排流程 | 输出 |
|---|---------|---------|---------|------|
| **S1** | `generate_proposals` | "帮我做个 MV"、用户选方案 | 音乐分析 → Agent 构思结构 → generate_image × N(角色/场景) → 汇总方案 | proposals [{ name, concept, structure, characters[], scenes[] }] |
| **S2** | `mv_planner` | 方案确定后 | Agent 校验时间轴 → 撰写每个 SHOT 的 treatment → 输出分镜列表 | SHOT 列表（纯文本） |
| **S3** | `mv_generator` | 分镜确定后 | Agent 将 treatment 转化为 video_prompt + frame_prompt → 输出结构化 JSON | shot JSON [{ shot_id, video_prompt, frame_prompt, start_time, end_time }] |
| **S4** | `video_generation` | "开始生成视频" | 逐 shot 调用 generate_video → 质量检测 → push_to_canvas | video_clips ready |
| **S5** | `revise_shot` | "Shot 3 重做" | 定位目标 → 理解修改意图 → 重新生成 → 推送到画布 | updated shot |
| **S6** | `export_mv` | "导出成片" | run_ffmpeg(concat) → run_ffmpeg(add_subtitles) → run_ffmpeg(convert) | final video |

**讨论点 6**：S2 和 S3 是否可以合并？
- S2（分镜规划）偏叙事层面，输出是文字描述
- S3（分镜生成）偏技术层面，输出是 JSON prompt
- **我的建议**：分开。因为用户可能想先看分镜规划（文字），确认后再转 JSON。也可能跳过规划直接要 JSON。

#### 创意分析节点 Skill（4 个模式）

每种模式对应一个独立 Skill，接收音乐分析结果 + 用户意图，输出 `mv_guide` JSON。

| # | Skill 名 | 对应文件 | 核心使命 | 时长约束 |
|---|---------|---------|---------|---------|
| **S7** | `creative_analysis_story` | `创意分析/prompts/generation/mv_story_mode_script_prompt.md` | 通过角色动作和故事推进让 MV 难忘，遵循现实物理规则 | 4-15s |
| **S8** | `creative_analysis_visions` | `创意分析/prompts/generation/mv_visions_script_prompt.md` | 把角色放入不可能存在的世界，每帧都是前所未见的视觉奇观 | 4-15s |
| **S9** | `creative_analysis_lipsync` | `创意分析/prompts/generation/mv_creative_script_lipsync_prompt.md` | 口型同步 MV，按 video_model 分段规则生成镜头序列 | 按模型 |
| **S10** | `creative_analysis_clone` | `创意分析/prompts/generation/mv_clone_mode_script_prompt.md` | 基于参考 MV 的视觉 DNA，重组到新歌时间轴上 | 4-15s |

**共同约束**：
- 输出纯 JSON，无 Markdown 包裹、无额外说明
- 顶层结构：`{ "mv_guide": { "style_guide", "md_stages", "mv_elements" } }`
- `md_stages` 列顺序：Time | Music Structure | Lyrics | Visual Description | Scene | Characters

#### 资产生成 Skill（3 个）

基于 `cinematic-prompt-generator` Skill，生成角色图和场景图的电影级图片提示词。

| # | Skill 名 | 定位 | 说明 |
|---|---------|------|------|
| **S11** | `asset_generation_character` | 角色图提示词生成 | 基于 9 个物理维度生成角色参考图提示词 |
| **S12** | `asset_generation_scene` | 场景图提示词生成 | 基于 9 个物理维度生成场景图提示词 |
| **S13** | `cinematic_prompt_generator` | 电影感提示词总 Skill | 适用于 Midjourney / SD 等工具，支持参考图反向分析 |

**9 维度执行流程**（S11/S12 共用）：
```
① 主题 → ② 景别 → ③ 构图 → ④ 焦段+虚实 → ⑤ 明暗骨架 → ⑥ 冷暖阶级
→ ⑦ 逻辑验证（质量门控） → ⑧ 相机+胶片 → ⑨ 神态+气质
→ 调用 T1 generate_image 执行生成
```

#### 视频生成 Skill 节点（4 个版本）

这些是 MV 生成流水线的核心执行节点，接收创意分析输出的 JSON，逐 shot 产出分镜脚本 JSON。

| Skill 名 | 对应文件 | 模型 | 说明 |
|----------|---------|------|------|
| mv-generation-一段式 | `视频skill/mv-generation-一段式/SKILL.md` | 4 模型 | infinite_talk / wan_video_2_7 / seedance_2_0 / kling_3_0_omni |
| mv-generation-二段式 | `视频skill/mv-generation-二段式/` | 4 模型 | planner + generator 双节点拆分 |
| mv-generation-sd2-一段式 | `视频skill/mv-generation-sd2-一段式/SKILL.md` | seedance_2_0 | SD2 专用单节点（story/visions/clone，4-15s） |
| mv-generation-sd2-二段式 | `视频skill/mv-generation-sd2-二段式/` | seedance_2_0 | SD2 专用双节点 |

**一段式的 Unwrap → Normalize → Dispatch 流程**：
1. Unwrap：解析 HumanMessage 包装或直接取 JSON
2. Normalize：mv_type 和 video_model 归一化
3. Dispatch：根据 `mv_type + video_model` 分发到对应 path 文件
4. 输出前自检：时间轴连续性、时长约束、JSON 结构

### L4 技能补充说明：创意分析节点

创意分析是 MV 生成流水线的核心节点，**每种模式对应一个独立 Skill**。它们接收音乐分析结果 + 用户意图，输出 `mv_guide` JSON（含 style_guide、md_stages 分镜表格、mv_elements 角色/场景列表）。

| Skill 名 | 对应文件 | 定位 | 核心使命 |
|----------|---------|------|---------|
| **S7 `creative_analysis_story`** | `创意分析/prompts/generation/mv_story_mode_script_prompt.md` | Story 模式 | 通过具体角色动作和故事推进，让 MV 令人难忘。遵循现实物理规则，角色在可识别的真实世界中行动 |
| **S8 `creative_analysis_visions`** | `创意分析/prompts/generation/mv_visions_script_prompt.md` | Visions 模式 | 把角色放入不可能存在的世界——每一帧都应该是前所未见的视觉奇观，由角色在异世界中的行动驱动，而非单纯的世界展示 |
| **S9 `creative_analysis_lipsync`** | `创意分析/prompts/generation/mv_creative_script_lipsync_prompt.md` | Lipsync 模式 | 口型同步 MV，按 video_model 分段规则生成镜头序列（infinite_talk / wan_video_2_7 / seedance_2_0 / kling_3_0_omni） |
| **S10 `creative_analysis_clone`** | `创意分析/prompts/generation/mv_clone_mode_script_prompt.md` | Clone 模式 | 基于参考 MV 的视觉 DNA，重组到新歌时间轴上 |

- 输出纯 JSON，无 Markdown 包裹、无额外说明
- 顶层结构：`{ "mv_guide": { "style_guide", "md_stages", "mv_elements" } }`
- `md_stages` 列顺序：Time | Music Structure | Lyrics | Visual Description | Scene | Characters
- 所有模式都内置 timeline normalization（story/visions/clone 为 4-15s 统一流程；lipsync 按模型分段规则独立）

### L4 技能补充说明：资产生成

资产生成 Skill 负责为方案阶段（generate_proposals）产出角色图和场景图。它们接收视觉风格描述 + 角色/场景设定，输出电影级图片生成提示词。

| Skill 名 | 来源 | 定位 | 说明 |
|----------|------|------|------|
| **S11 `asset_generation_character`** | `Tunee Skill库/cinematic-prompt-generator/` | 角色图提示词生成 | 基于 9 个物理维度（主题/景别/构图/焦段/光影/色彩/逻辑/胶片/神态）生成角色参考图提示词 |

| **S13 `cinematic_prompt_generator`** | `Tunee Skill库/cinematic-prompt-generator/SKILL.md` | 电影感图片提示词总 Skill | 适用于 Midjourney、Stable Diffusion 等工具的通用提示词生成，支持参考图反向分析 |

**执行流程**（S11/S12 共用）：
```
输入：角色/场景描述 + 视觉风格
  → 逐模块查阅 references/01-09.md 词库
  → ① 主题 → ② 景别 → ③ 构图 → ④ 焦段+虚实 → ⑤ 明暗骨架 → ⑥ 冷暖阶级
  → ⑦ 逻辑验证（质量门控，必须执行） → ⑧ 相机+胶片 → ⑨ 神态+气质
  → 输出：完整的图片生成 prompt
  → 调用 T1 generate_image 执行生成
```

### L4 技能补充说明：视频生成 Skill 节点

视频 Skill 是 MV 生成流水线的核心执行节点，**接收创意分析输出的 JSON，逐 shot 产出分镜脚本 JSON**。

| Skill 名 | 对应文件 | 定位 | 支持的模型 |
|----------|---------|------|-----------|
| **mv-generation-一段式** | `视频skill/mv-generation-一段式/SKILL.md` | 单节点完整版 | infinite_talk / wan_video_2_7 / seedance_2_0 / kling_3_0_omni |
| **mv-generation-二段式** | `视频skill/mv-generation-二段式/` | 双节点拆分（planner + generator） | 同一模型表 |
| **mv-generation-sd2-一段式** | `视频skill/mv-generation-sd2-一段式/SKILL.md` | Seedance 2.0 专用单节点 | seedance_2_0（4-15s） |

**一段式的 Unwrap → Normalize → Dispatch 流程**：
1. Unwrap：解析 HumanMessage 包装或直接取 JSON
2. Normalize：mv_type 归一化（lip_sync → lip-sync）、video_model 归一化（5 种模型 key 映射）
3. Dispatch：根据 `mv_type + video_model` 分发到对应 path 文件，每个 path 是独立的创作指导
4. 输出前自检：时间轴连续性、时长约束、JSON 结构

## 六、关键设计决策

### 决策 1：LLM 内部能力不注册为 Tool，M1 也不注册为 Tool

| 不算 Tool | 原因 |
|-----------|------|
| 意图识别 | LLM 分析用户消息，纯推理 |
| 文字/JSON 生成 | LLM 本身就是文字生成器 |
| 模型选择 | 查表 + 推理，不涉及外部调用 |
| 时间轴校验 | 数学计算 |
| 前置条件检查 | 读 State → 判断 |
| 回退决策 | 根据错误类型选择策略 |
| **M1 activate_workspace** | 是前端模式切换回调，不是外部 API |

**好处**：减少 Tool 数量 → 上下文更干净 → 模型调用更准确。

### 决策 2：push_to_canvas 是唯一的前端同步通道

画布的任何变更，**只通过 `push_to_canvas` 触发**。不直接在 Tool 返回值里塞 DOM 操作。

```
generate_image 完成
  → 返回 { image_url, seed, ... }
  → Agent 决定推送到画布
  → push_to_canvas("asset_created", "visual_assets", { url, metadata })
  → 前端 EventBus 分发 → 对应区域渲染
```

**好处**：工具是纯函数，不依赖前端。换前端不用改 Tool。

### 决策 3：Page 是画布的容器，不是 Agent 的概念

Page 是前端画布的组织单位。Agent 只知道 "push_to_canvas 到某个 region"，但 region 属于哪个 Page，是前端的事。

但 Agent 需要知道当前用户在哪个 Page 上工作，所以：
- `list_pages` 让 Agent 知道有哪些 Page
- `switch_page` 让 Agent 能切到用户指定的 Page
- Agent State 中维护 `current_page_id`

### 决策 4：Skill 的注册方式

Skill 不应该像 Tool 一样注册到 LLM 的 tools 数组。原因：
- Skill 是编排逻辑，参数复杂
- LLM 不需要"看到" Skill 的内部步骤
- Skill 应该被封装为 Agent 的一个 Node（LangGraph）或一个 Method

**建议的注册方式**：

```python
# Mode 切换注册到 Agent 框架（作为前端回调）
mode_actions = {
    "activate_workspace": activate_workspace,
    "deactivate_workspace": deactivate_workspace,  # V2
}

# Tool 注册到 LLM（LLM 可以看到并决定调用）
tools = [generate_image, generate_video, run_ffmpeg, evaluate_quality,
         web_search, read_asset, analyze_music, lip_sync, face_swap]

# Skill 注册到 Agent 框架（作为可编排的流程节点）
skills = {
    # 创意分析节点（4 个模式）
    "creative_analysis_story": creative_analysis_story_skill,
    "creative_analysis_visions": creative_analysis_visions_skill,
    "creative_analysis_lipsync": creative_analysis_lipsync_skill,
    "creative_analysis_clone": creative_analysis_clone_skill,

    # 资产生成（3 个）
    "asset_generation_character": asset_generation_character_skill,
    "asset_generation_scene": asset_generation_scene_skill,
    "cinematic_prompt_generator": cinematic_prompt_generator_skill,

    # 视频生成（4 个版本）
    "mv_generation_single_node": mv_generation_single_node_skill,
    "mv_generation_two_phase": mv_generation_two_phase_skill,
    "mv_generation_sd2_single": mv_generation_sd2_single_skill,
    "mv_generation_sd2_two_phase": mv_generation_sd2_two_phase_skill,

    # 编排型 Skill（6 个）
    "generate_proposals": generate_proposals_skill,
    "mv_planner": mv_planner_skill,
    "mv_generator": mv_generator_skill,
    "video_generation": video_generation_skill,
    "revise_shot": revise_shot_skill,
    "export_mv": export_mv_skill,
}

# 画布操作注册到 Agent（作为内部回调）
canvas_actions = {
    "push_to_canvas": push_to_canvas,
    "open_page": open_page,
    ...
}
```

### 决策 5：原子能力的输入输出设计原则

1. **输入尽量结构化**：用明确的字段，不要靠 LLM 猜
2. **输出包含元数据**：不仅返回结果，还返回 model_used, seed, cost 等
3. **异步任务统一处理**：generate_video 是异步的，需要 task_id + polling 机制
4. **错误类型标准化**：{ error_code, message, retryable }，方便回退逻辑

## 七、完整调用示例：从零到完整 MV

```
用户: "帮我为《晴天》做一支 MV" [上传: 晴天.mp3]

Step 0: Agent 识别意图 = mv_creation
  → 模式层: M1 activate_workspace("mv_creation", initial_layout="default")
    → UI 从全屏对话 → 360px 对话 + 70% 画布
    → 初始化: 创建 Page_1，设置默认 region 布局
    → 返回: { workspace_id: "ws_1", page_id: "page_1" }

Step 1: Agent 识别意图 = mv_creation
  → 感知层: P2 analyze_music("晴天.mp3")
    → 返回: { bpm: 128, key: "G", structure: ["Intro","Verse","Chorus","Bridge","Outro"] }
  → 画布层: C1 push_to_canvas("music_analysis", "music_area", data)
  → 回复: TextReply("这首歌 BPM 128，G 大调，结构为 Intro→Verse→Chorus→Bridge→Outro")

Step 2: 用户: "开始吧"
  → Agent 调用 Skill: S1 generate_proposals
    → 内部: LLM 构思 3 个方案（叙事结构 + 视觉风格）
    → 工具: T1 generate_image × 6 (3 方案 × 角色 + 场景)
    → 画布: C1 push_to_canvas("proposal_generated", "proposal_area", data)
  → 回复: ProposalCard × 3

Step 3: 用户: "选方案 A"
  → Agent 更新 State
  → 画布: C1 push_to_canvas("proposal_selected", "proposal_area", { id: "A" })
  → 回复: TextReply("好的，开始为方案 A 规划分镜")

Step 4: 用户: "生成分镜"
  → Agent 调用 Skill: S2 mv_planner → 输出文字分镜
  → 画布: C1 push_to_canvas("storyboard_ready", "storyboard_area", data)
  → Agent 调用 Skill: S3 mv_generator → 输出 JSON
  → 画布: C1 push_to_canvas("shots_generated", "timeline", data)
  → 回复: TextReply("8 个镜头已规划完成，要生成视频吗？")

Step 5: 用户: "生成视频"
  → Agent 调用 Skill: S4 video_generation
    → 逐 shot: T2 generate_video(frame_url=分镜图, duration=时长)
    → 每完成一个: C1 push_to_canvas("video_clip_progress", "timeline", data)
    → T4 evaluate_quality 自动质检
    → 全部完成: C1 push_to_canvas("video_clips_ready", "timeline", data)
  → 回复: TextReply("MV 生成完成！可以预览或导出")

Step 6: 用户: "Shot 3 的脸歪了，重做"
  → Agent 调用 Skill: S5 revise_shot
    → P1 read_asset(Shot_3.frame_url) → 理解当前状态
    → LLM 分析用户意图 → 修改 prompt
    → T1 generate_image(new_prompt, asset_type="storyboard")
    → T2 generate_video(new_frame_url)
    → T4 evaluate_quality → score 85 → 通过
    → C7 update_element("Shot_3", { video_url: new_url })
  → 回复: AssetCard(新分镜图 + 新视频)

Step 7: 用户: "导出成片"
  → Agent 调用 Skill: S6 export_mv
    → T3 run_ffmpeg(operation="concat", inputs=[所有shot])
    → T3 run_ffmpeg(operation="add_subtitles", ...)
    → T3 run_ffmpeg(operation="convert", format="mp4")
  → 回复: AssetCard(最终视频 + 下载按钮)
```

## 八、设计决策确认

> 以下决策已于 2026-04-26 确认，讨论稿转为执行稿。

### Q1：analyze_music 拆不拆？

**决策：本次不拆，V2 再考虑拆分。**

MV 场景中几乎总是需要全部信息（BPM + 调性 + 情绪曲线 + 歌词结构），一次调用拿全信息，减少延迟。后续如果某个子能力被高频单独调用，再拆为独立 Tool。

### Q2：lip_sync 和 face_swap 独立还是内嵌？

**决策：独立为原子能力（T6 / T7）。**

理由：
1. 模型能力在快速变化，今天内置的明天可能收费或下线
2. 独立的 tool 可以随时切换底层实现，不依赖视频模型
3. 用户可以单独调用（比如"帮我把这个镜头的口型对上"）

### Q3：画布操作是否全自动化？

**决策：保留为 Agent 可控能力（L2 画布层）。**

理由：
1. Tool 完成不代表要推送到画布（如 evaluate_quality 可能只在对话区展示）
2. Agent 需要推理推送到哪个 region
3. 前端可能有多个 Page，需要 Agent 明确推送目标

### Q4：是否需要 compose_prompt 原子能力？

**决策：目前不需要。**

LLM 本身就是最好的 prompt 生成器。专门的 prompt compose 工具反而增加复杂度。prompt 模板逻辑可以在 Skill 层封装。

### Q5：多 Page 场景下 Agent State 如何管理？

**决策：方案 A — Agent 维护 Map<page_id, State>。**

最灵活，Agent 可以在不同 Page 之间做交叉操作（比如"把 A 方案的角色图用到 B 方案里"）。每个 Page 独立维护自己的 pipeline_stage、assets、shots 等。

配合 **M1 `activate_workspace`** 使用：第一次激活画布时创建 Page_1，后续用户说"再做一首"时开新 Page。
