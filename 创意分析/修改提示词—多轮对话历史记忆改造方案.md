# 修改提示词 — 多轮对话历史记忆改造方案

> 版本：v1.0
> 日期：2026-04-17
> 适用范围：三个修改提示词（story_mode/visions、lipsync、clone_mode）

---

## 1. 改造目标

为三个 MV 脚本修改节点引入最多 5 轮对话历史，每轮同时携带用户修改指令（`user_modification`）和该轮产出的 mv_guide 快照（`ori_mv_guide`）。让模型理解完整对话轨迹——用户改了什么、为什么改、拒绝了什么——从而做出更精准的增量修改决策。

同时新增 **Revert 模式**，使节点能够直接恢复上一版 mv_guide。

---

## 2. 历史数据结构

### 2.1 新增输入字段

在三个 `*_memory.md` 提示词的 Canonical fields 中新增：

```
`history` (array, optional, default [])
```

### 2.2 单条记录结构

```json
{
  "user_modification": "把整体色调改成日系胶片感",
  "ori_mv_guide": {
    "md_stages": "| 时间段 | 音乐结构 | 歌词 | 画面描述 | 场景 | 角色 |\n|---|---|---|---|---|---|\n...",
    "mv_elements": {
      "characters": [ ... ],
      "scenes": [ ... ]
    }
  }
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `user_modification` | string | 用户在该轮的修改指令原文（即当时的 `user_modification`） |
| `ori_mv_guide` | object | 该轮输出的 mv_guide 对象，**裁剪掉 `style_guide`**（只读字段，始终从当前调用的 `ori_mv_guide` 复制） |

### 2.3 数组规则

- 按时间顺序排列，**最新一轮在数组末尾**
- 最大容量 **5 条**，超出时从头部丢弃最旧条目（FIFO）
- 第一轮修改时 `history = []`

---

## 3. 调用方职责

调用方（上游节点）负责构建和维护 `history`，提示词本身只读不写。

### 3.1 构建新条目

每次修改调用返回成功后：

```python
entry = {
    "user_modification": current_user_modification,
    "ori_mv_guide": {
        "md_stages": response["mv_guide"]["md_stages"],
        "mv_elements": response["mv_guide"]["mv_elements"]
    }
}
history.append(entry)
if len(history) > 5:
    history = history[-5:]
```

### 3.2 Revert 模式

- Revert 调用后，**不**将结果追加到 history
- history 保持不变，后续新修改基于回退后的状态继续

### 3.3 历史重置

用户开始全新 MV 生成时，`history` 初始化为 `[]`。

## 4. 提示词改动

三个 memory 版提示词各自新增以下内容（结构一致，模式特定约束保持各自差异）：

### 4.1 Input Normalization

- Canonical fields 列表新增 `history` 字段声明
- Defaults 表格新增：`history` missing → default `[]`

### 4.2 新增 Section: Multi-Round History

- **history Array Structure** — 字段定义、JSON 结构、类型表
- **History-Aware Modification** — 模型如何使用历史：
  1. 理解对话轨迹（遍历 user_modification）
  2. 避免重复已被否决的改动（指令冲突时以当前为准）
  3. 增量构建（history[-1] 作为上下文参照）
  4. 冲突检测（当前指令覆盖历史意图）
- **Revert Mode** — 固定字符串触发，直接返回 history 最后一轮的 ori_mv_guide + style_guide

### 4.3 Execution Order 调整

在原有执行顺序中插入历史解析步骤：

```
1. Extract and normalize payload
2. Parse history — if Revert mode and history non-empty, return history[last].ori_mv_guide + style_guide and exit
3. Parse ori_mv_guide
4. If history non-empty, understand conversation trajectory
5. Classify modification scope
6. Apply modifications + Timeline Repair
7. Rebuild mv_elements
8. Copy style_guide
9. Run Output Gate
10. Return JSON
```

### 4.4 各文件对应关系

| 原有文件 | 新增文件 | 模式 |
|---|---|---|
| `mv_creative_script_modification_prompt.md` | `mv_creative_script_modification_prompt_memory.md` | story_mode / visions |
| `mv_creative_script_lipsync_modification_prompt.md` | `mv_creative_script_lipsync_modification_prompt_memory.md` | lip_sync |
| `mv_clone_mode_modification_prompt.md` | `mv_clone_mode_modification_prompt_memory.md` | clone_mode |

## 5. Token 控制

### 5.1 快照裁剪

每个快照仅含 `md_stages` + `mv_elements`，裁剪 `style_guide`（始终从 `ori_mv_guide` 原样复制）。

### 5.2 Token 预算估算

以典型 MV（8 行 md_stages，2 角色，2 场景）为例：

| 内容 | 单轮 token | 5 轮上限 |
|---|---|---|
| `user_modification` | ~15 | ~75 |
| `ori_mv_guide.md_stages` | ~200 | ~1,000 |
| `ori_mv_guide.mv_elements` | ~80 | ~400 |
| **单轮合计** | **~295** | **~1,475** |

占总预算（32K-128K context window）比例约 1.2%–4.7%。

### 5.3 极端情况

如果 mv_guide 异常大（>20 行），调用方可选择仅保留最近 3 轮或对 `md_stages` 做摘要式压缩。

---

## 6. 验证策略

### 6.1 回归检查

改造后，以 `history: []` 调用时，行为应与原文件完全一致。

### 6.2 新增用例

| 用例 | history | 期望行为 |
|---|---|---|
| 第一轮修改 | `[]` | 与原文件行为一致 |
| 第二轮修改 | `[第1轮]` | 理解第1轮指令，正确增量修改 |
| 第五轮修改 | `[1-4轮]` | 正确理解完整对话轨迹 |
| 第六轮修改 | `[2-5轮]`（第1轮已丢弃） | 不依赖被丢弃的条目，正常工作 |
| Revert | `"Revert to previous round"`, `[≥1轮]` | 输出 = history 最后一轮的 ori_mv_guide + style_guide |
| Revert 空历史 | `"Revert to previous round"`, `[]` | 返回 error: "no previous round to revert to" |
| 意图反转 | `[{"modification": "改成暖色"}, ...]`, 当前指令"太暖了改回来" | 以当前指令为准 |

## 7. 实施说明

### 7.1 文件策略

- **保留原文件不变**：3 个 `_memory.md` 是新文件，不修改现有文件
- 上游节点可选择使用 memory 版本或原版，互不干扰

### 7.2 接入步骤

1. 将 3 个 `_memory.md` 文件部署到提示词服务
2. 上游 Modification Judge 节点增加 history 构建逻辑（见第 3 节）
3. 先接入 `history: []` 验证与原行为一致
4. 逐步引入真实历史数据，从 1 轮开始验证
5. 全量上线后监控 token 消耗和输出质量
