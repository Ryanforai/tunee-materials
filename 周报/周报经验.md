# 周报写作经验

## 核心原则

**周报 = 成果展示，不是流水账。** 领导关心的是"你带来了什么改变"，不是"你做了多少事"。

---

## 写作框架

### 1. 本周概要（1-2 句话）

一句话总结本周最大成果 + 整体状态（正常 / 有风险 / 需支持）。

### 2. 核心成果（按项目分组）

公式：**[项目] 动作 + 量化指标 + 业务价值**

每个成果回答三个问题：
- **做了什么？**（具体动作）
- **效果如何？**（量化指标）
- **为什么做？**（业务影响 / 解决的问题）

### 3. 问题与应对

问题 + 当前状态 + 应对策略。展示你在主动处理，不是等别人解决。

### 4. 下周重点

1-3 条，按优先级排序，写清预期产出。

### 5. 需要的支持

明确的资源 / 决策 / 协作请求。没有就写"无"。

---

## 写法对比

### ❌ 流水账写法（只写动作）

```
- 修改了 mv_creative_script_lipsync_prompt.md
- 删除了 mv-generation 目录下的 30 个文件
- 新增了数据分析目录
- 整理了 Skill 目录结构
```

### ✅ 成果导向写法（动作 + 指标 + 价值）

```
- 统一 Lipsync 模式时间线处理为 Step 0-4 标准流程，
  消除 5 种模式的流程差异，后续新增模式开发时间预计缩短 40%+
- 清理旧版 Skill 冗余目录 ~30 个文件，消除新旧版本混淆，
  团队按版本号明确引用路径
- 建立 MV 生成数据分析体系，首次系统化评估生成质量，
  为后续优化提供量化依据
```

## 关键技术动作的翻译

| 技术动作 | 领导看得懂的说法 |
|---|---|
| "优化了 prompt 模板" | "Agent 提示词优化使任务完成率从 X% 提升到 Y%，支持 Z 个新流程上线" |
| "重构了 timeline 处理" | "统一 N 种模式的执行流程，减少维护成本，后续新增模式开发时间缩短 X%" |
| "整理了 Skill 目录" | "完成 Skill 归档，消除冗余文件，团队协作更清晰，避免选择歧义" |
| "新增了 memory 文件" | "支持多轮对话上下文保持，用户修改时不用反复重复需求" |
| "建了数据分析体系" | "首次建立生成质量量化评估手段，优化效果可衡量" |

## 常见错误

| 错误 | 说明 |
|---|---|
| 按时间顺序罗列 | "周一做了A，周二做了B" — 领导要的是结果，不是日程表 |
| 只写动作不写结果 | "花了 3 小时优化 prompt" — 没说效果 |
| 用技术术语不解释 | "重构了 timeline normalization" — 没说解决了什么问题 |
| 没有下周计划 | 只看过去，不展望未来 |
| 每周格式不一致 | 领导无法快速扫描你的周报 |
| 过度强调过程 | 做了什么不重要，做成了什么才重要 |

## 自检清单

写完周报后问自己：

- [ ] 每条成果是否说明了"为什么做"？
- [ ] 是否有量化指标（数字 / 百分比 / 对比）？
- [ ] 技术术语是否翻译成了业务语言？
- [ ] 是否有下周计划？
- [ ] 格式是否和上周保持一致？
- [ ] 能否在 3 分钟内读完？

---

## 每周新建流程

1. 在 `周报/` 目录下新建 `YYYY-MM-DD~YYYY-MM-DD.md`（周一至周日）
2. 复用本文件中的框架
3. 完成后 `git add` + `git commit`
4. 更新 `CLAUDE.md` 中的现有周报列表
